# Do Eval

Do Eval is a local-first evaluation harness for coding agents. Use it to run
the same tasks against Pi, Codex, or another registered harness; verify the
resulting workdir; score the session; and compare agent behavior over time.

It is built for agent extensions, Codex plugins, skill packs, and runtime
profiles where the important question is: did the agent do better work on the
same coding task than it did before?

Do Eval gives you:

- resettable trials with source-controlled task files and scaffolds
- suites for smoke, regression, and benchmark coverage
- Trial, Regression, and Bench launch modes
- deterministic verification plus optional LLM judge scores
- profile comparisons for models, harnesses, skills, plugins, and setup layers
- live run status, reports, and history in the terminal TUI
- a lower-level `runEval` API for custom tooling

## Run It

If you are developing Do Eval itself from this checkout:

```bash
cd /path/to/pi-do-eval
npm install
npm run build
bun cli/index.ts tui --project /path/to/project/eval
```

If you are inside an eval project generated by `do-eval init`:

```bash
cd /path/to/project/eval
npm install
npm run eval:list
npm run eval
```

If you are elsewhere and already have `do-eval` on your path:

```bash
do-eval tui --project /path/to/project
do-eval regression small --project /path/to/project/eval
```

For the full command map, read [Running Do Eval](docs/running.md).

## Start Here

Run the scaffold from the root of the repo you want to evaluate:

```bash
npx do-eval init
cd eval
npm install
npm run eval:list
npm run eval
```

The scaffold creates this layout:

```text
eval/
  package.json
  tsconfig.json
  eval.config.ts
  plugins/
    <project>.ts
  trials/
    example/
      trial.yaml
      task.md
  suites/
    small.yaml
```

The generated example is a smoke test for the harness, not a meaningful eval
yet. To make it real:

1. Replace `trials/example/task.md` with a task that represents work you care
   about.
2. Update `plugins/<project>.ts` so `verify()` and `scoreSession()` describe
   success for that task.
3. Add more trial directories under `trials/`.
4. Group trial variants into suites under `suites/`.
5. Run Trial while authoring, Regression for one profile over time, and Bench
   when comparing profiles.

Common commands:

```bash
npm run eval:list
npm run eval:trial -- example --variant default
npm run eval:regression
npm run eval:bench
npm run eval
```

## TUI

`do-eval tui` is the primary terminal UI for Launch and Runs. Historical
reports, trends, and comparisons belong in the web viewer.

```bash
npm run eval
do-eval tui
do-eval tui --project ~/sandbox/pi-proof
do-eval tui --project ~/sandbox/pi-proof --port 4242
```

Use `npm run eval` inside an eval project. Use bare `do-eval` only when the
package is installed globally or your shell `PATH` includes the local
`node_modules/.bin` directory.

The bottom status bar shows active run progress and the matching web viewer
state. By default the TUI starts or reuses the local web viewer and shows
`Web starting http://localhost:4242`, `Web http://localhost:4242`, or
`Web unavailable`. Pass `--no-web` if you only want the terminal UI.

Useful keys:

| Key | Action |
| --- | --- |
| `q` | Quit at the project list |
| `Ctrl+C` | Quit from anywhere |
| `Esc` | Back or cancel |
| `Enter` | Open the selected item |
| `j/k` or arrows | Move selection |
| `b` | Run Bench |
| `g` | Run Regression |
| `t` | Run Trial |
| `c` | Runs screen |
| `s` | Suites screen |
| `T` | Trials screen |
| `R` | Refresh |
| `e` | Edit the selected suite YAML |
| `d` | Delete the selected suite after confirmation |
| `Tab` or arrows | Switch run-detail tabs |

## Web Viewer

The web viewer is the historical view for completed runs, reports, trends, and
comparisons. `do-eval tui` starts it automatically by default; run it directly
when you want the browser view without the terminal UI.

```bash
do-eval ui --project . --port 4242
do-eval ui-dev --project . --port 4242
```

The viewer and TUI share the same run directories, suite YAML files, and
per-user project registry. Project registry files live under the historical
`pi-do-eval` config directory:

- `$PI_DO_EVAL_CONFIG_HOME/pi-do-eval/projects.json` if set
- `$XDG_CONFIG_HOME/pi-do-eval/projects.json` if set
- `~/.config/pi-do-eval/projects.json` otherwise
